# 設計判断ログ

このプロジェクトで意図的に下した判断と、その理由を残す。「なぜそうなっているか」の問いに最初に当たるべきファイル。

---

## 1. Vanilla JS を採用（React / Vue 等を不採用）

**判断**: フレームワーク無し、ビルドステップ無しの素のJSで実装。

**理由**:
- 画面は4つだけ（スタート/設定/出題/終了）で、状態数も少ない
- Cloudflare Pages に「Build command 空欄」でそのまま置ける
- Service Worker のキャッシュ管理がシンプル（ハッシュ付きJSのバージョン管理不要）
- 1ファイル(app.js)約280行で完結
- ST業務的に追加機能の頻度は低い想定

**いつ覆すか**: 結果記録・統計画面など複雑な状態管理が必要になったら React + Vite で書き直す価値あり。それまでは Vanilla で十分。

---

## 2. 音声は VOICEVOX 事前生成 wav（Web Speech はフォールバック）

**判断**: VOICEVOX エンジン (`No.7 アナウンス` / styleId=30) で142語を事前生成し、`audio/[category]/[id].wav` として同梱。再生は HTML `<audio>` で行い、取得失敗時は Web Speech API にフォールバック。

**経緯**:
1. 当初は Web Speech API のみで運用（iPad Kyoko Enhanced 想定）
2. iPad voice の精緻化検討中、もう一段上を狙って AivisSpeech を試作
3. AivisSpeech のデフォルト声 (まお/コハク) は**アニメ調**で医療現場には不適合
4. AivisHub 上位モデルの「Anneli」に**声優・山村響さんの無断学習問題**が判明、医療施設の公式アプリでは使用不可と判断
5. ライセンスが公式で一元管理される **VOICEVOX に切替**、ナレーション特化の `No.7 アナウンス` を選定

**理由**:
- 医療施設アプリとして**学習元のクリアランスがプラットフォームレベルで保証**されている必要がある
- VOICEVOX 各キャラの利用規約は公式サイトで明示、商用利用可
- `No.7 アナウンス` は名前通りアナウンス向けに調整された、感情の薄い明瞭な大人女性声
- 事前生成 wav なら端末差・iOS Safari の TTS タイミング問題も発生しない
- フォールバックで Web Speech を残しておけば、新規追加カードで wav 未生成のケースも黙って自動対応

**サイズ感**:
- wav 142ファイル合計 4.9MB（平均 35KB）
- 既存の画像 7.8MB と合わせて約13MB、iOS の SW キャッシュ上限 ~50MB に対し余裕

**運用**:
- 生成は `node scripts/generate-audio.mjs`（差分のみ） / `--force`（全件再生成）
- VOICEVOX アプリ起動中（`http://127.0.0.1:50021`）が前提
- アプリ追加後は `sw.js` の `CACHE_NAME` を上げる

**いつ覆すか**:
- ST から「No.7 ではアクセントが不自然」フィードバックが特定の語で出たら、VOICEVOX GUI でアクセント手動補正して該当語のみ再生成
- 話者を変えたくなったら `scripts/generate-audio.mjs` の `SPEAKER_ID` を変えて `--force` 再生成

## 3. 画像最適化：512px / PNG palette / sharp

**判断**: GPT Image 2.0 で生成した画像を sharp で 512x512 max、PNG palette mode、compressionLevel: 9 で圧縮。

**実績**:
- 平均 851KB → 55KB（93%削減）
- 全画像 118MB → 7.8MB
- 視覚的劣化はほぼ無し（拡大しないなら気づかないレベル）

**判断のポイント**:
- WebP も検討したが、iOS 14+ で対応とはいえ古いiPad考慮で PNG 維持
- 512px は仕様書記載の「長辺512〜768px」の最小値（カード枠が最大520px なので十分）
- palette mode はイラスト系画像と相性が良い

**いつ覆すか**: 写真系画像を扱うようになったら palette mode を切る。WebP に変えるなら画像参照側のフォールバック対応も必要。

## 4. 画像先読み（次の2枚を裏で取得）

**判断**: `renderCurrentCard()` 内で `preloadUpcomingImages()` を呼び、`new Image()` で next + 1 と next + 2 の画像URLをセット。

**理由**:
- 答え表示中（ユーザーが画像を見ている時間）に裏でロードできる
- ブラウザのHTTPキャッシュに乗るので、次の `img.src = ...` 設定時に即表示
- Cloudflare CDN 経由でも体感「ほぼ瞬時」になる
- PWA キャッシュとも自然に共存（SW がキャッシュ済みなら即返却）

---

## 5. ヒントモード：モーラ単位 + タップで個別開示

**判断**: 拗音(ゃゅょ)・促音(っ)を伴う文字は2文字を1ユニットとして扱い、各ユニットを独立 `<span>` でレンダー。マスクされた `〇` をタップで開示。

**例**:
- 「しょうぼうしゃ」→ `[しょ][〇][〇][〇][〇]` （5ユニット、7文字ではない）
- 「ぎゅうにゅう」→ `[ぎゅ][〇][〇][〇]` （4ユニット）
- 「りんご」→ `[り][〇][〇]`

**理由**:
- ST 業務上、訓練単位は「モーラ」が自然（「し」+「ゃ」を別カウントは違和感）
- 「しゃ」単体で1ユニット表示することで意味のあるヒントになる
- 1文字単独タップで「ゃ」だけ出てくる挙動を避けられる

**実装**:
- 検出正規表現: `/[ゃゅょぁぃぅぇぉっャュョァィゥェォッ]/`
- 各ユニット毎に `<span class="hint-char masked">` を生成、クリックで `revealed` に切替

## 6. PWA / Service Worker でオフライン化

**判断**: Service Worker で全アセット（HTML/CSS/JS/data/142画像 = 約8MB）を install 時にプリキャッシュ。`cache-first` + `network fallback`。

**理由**:
- iPad で「ホーム画面に追加」→ 機内モード・院内Wi-Fi切断時でも動作可能にする
- ST業務環境（医療施設）はネット接続が不安定なことがある
- iOS の SW キャッシュ上限 ~50MB に対し 8MB は十分余裕
- Web Speech API は iOS 内蔵なのでオフラインで動く（クラウド呼び出し不要）

**キャッシュ戦略**:
- `install`: `SHELL_ASSETS` + cards.json から動的生成した画像URLを `cache.addAll`
- `activate`: 古いキャッシュを削除（`CACHE_NAME` バージョン更新時）
- `fetch`: キャッシュ優先、ネットワーク取得時もキャッシュに追加、`navigate` 失敗時は `./` にフォールバック

**重要**: アプリ更新時は `sw.js` の `CACHE_NAME` を `v1` → `v2` のように上げる。さもないと古いキャッシュがそのまま使われる。

## 7. Cloudflare Pages（Vercel/Netlify 不採用）

**判断**: Cloudflare Pages を採用。

**理由**:
- 無料枠が広い（無制限のリクエスト数、500ビルド/月）
- 仕様書記載の選択
- GitHub 連携で自動デプロイ
- HTTPSが自動で付くので PWA の Service Worker 要件を満たす
- 日本国内のエッジが多くて速い

**運用**:
- `main` ブランチへの push で自動再デプロイ（30秒〜1分）
- 環境変数・サーバーサイド処理は使っていない（純静的サイト）

---

## 8. デザイン方針：清潔・落ち着き重視

**判断**: 青ベース（#2563eb）、白カード + 薄いグレー背景、控えめなアニメーションのみ。

**理由**:
- 失語症リハの患者層は**高齢者が多い**
- 派手な動き・色は集中を妨げる、視覚過敏の人には逆効果
- 「医療系プロダクト」らしい清潔感が必要
- iPad で患者と並んで使うので、画面の主役は **カード画像** であるべき（UIは脇役）

**やらないこと**:
- ジャンプするアニメーション
- 派手な BGM・効果音
- ゲーム的なスコア表示・ランキング
- アバター・キャラクター演出

**やってもOK**:
- 軽いフェード遷移
- ヒント開示時のサブトルなポップアニメ（実装済）
- 完了画面の「🎉」程度（実装済）

## 9. データ形式：単一 JSON、カテゴリは別ファイル

**判断**: 全カード情報を `data/cards.json` 1ファイルに集約。カテゴリ定義は `data/categories.json` で分離。

**理由**:
- 142件程度なら 1 ファイルで十分扱える（圧縮後 ~17KB）
- カテゴリ追加・削除はカテゴリファイルだけ触れば良い
- カード追加時の競合衝突を最小化（複数のSTが将来編集するケース想定）

**フィールド**:
- `id`: `[category]_[name]` 形式、画像ファイル名と一致必須
- `japanese_label`: 画面表示用（ひらがな/カタカナ）
- `reading`: TTS 用（ひらがな統一）
- `familiarity`: `high` / `mid` / `low`（TLPA等の臨床基準準拠）
- `english_prompt`: 画像再生成時の参考

## 10. 出題設定：親密度別問題数スライダー（総問題数スライダーを廃止）

**判断**: 「総問題数スライダー + 親密度チェックボックス（含む/含まない）」の組み合わせを廃止し、**「親密度3種それぞれに問題数スライダーを1本ずつ」+ 合計自動計算** に変更。加えて `やさしい中心 / バランス / むずかしめ / クリア` のプリセットボタンを用意。

**理由**:
- ST の実務ニーズは「軽度失語には高頻度中心」「回復期は中頻度多め」「維持期は低頻度混合」のように**親密度ごとに問題数を細かく指定したい**
- 旧UI（チェックボックス + 総問題数）ではパーセント計算が STの頭の中で必要で、直感的でなかった
- 直接カウント指定なら「高5 / 中3 / 低2」のような指示が一目で可視化・調整できる

**実装ポイント**:
- 各スライダーの max はカテゴリ選択に応じて動的更新（`getAvailableByFamiliarity()`）
- カテゴリ変更で最大値が下がったら現在値を自動クランプ
- 親密度別に対応カードがゼロの場合、スライダーは disabled + `.is-empty` で視覚的に沈める
- サンプリングは親密度ごとにシャッフル→指定数抽出→全体を再シャッフル（`startQuiz()`）
- シャッフル OFF時は `high→mid→low` の順で出題

**プリセット**:
- easy: 10/0/0（軽度失語・スクリーニング向け）
- balanced: 4/4/2（一般的な混合、旧デフォルトの総10問と合致）
- hard: 2/4/4（回復期後半・維持期向け）
- clear: 0/0/0

**いつ覆すか**: STから「プリセットが実務のパターンと合わない」フィードバックが出たら、頻用の親密度分布を洗い出して追加または差替え。
